## Problem

Website leads coming through `submit-lead` get codes like `ADV-2026-004`, but the rest of the CMS uses the standard format `MA2600001` (month-code + 2-digit year + 5-digit sequence) generated by the `generate_traveller_code` DB trigger on `public.leads`.

Root cause: `supabase/functions/submit-lead/index.ts` explicitly calls `supabase.rpc("generate_adv_traveller_code")` and passes the result into the insert, bypassing the trigger.

## Fix (single file)

`supabase/functions/submit-lead/index.ts`:
- Remove the `supabase.rpc("generate_adv_traveller_code")` call and its error branch.
- Insert with `traveller_code: ""` so the existing `trg_leads_traveller_code` BEFORE INSERT trigger generates the code in the standard format — same path the CRM "Add Lead" button uses.
- Keep `.select("id, traveller_code").single()` so the response still returns the generated code.

No frontend, schema, or trigger changes needed. The existing test lead row (`ADV-2026-004`) can be left as-is or deleted manually — it does not affect future submissions.

## Verification

- Re-deploy `submit-lead`.
- POST a test payload via curl; expect `traveller_code` in the response to match `^[A-Z]{1,2}\d{2}\d{5}$` (e.g. `MA2600007`).
- Confirm the row in Lead Management shows the same standard-format code.
